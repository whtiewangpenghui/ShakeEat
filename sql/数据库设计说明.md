## 数据库设计说明

### 数据库整体怎么理解

`shakeeat` 现在只有 5 张核心表, 可以按 3 个业务块理解:

| 业务块 | 相关表 | 作用 |
| :-- | :-- | :-- |
| 菜单库 | `food_item`, `food_tag`, `food_item_tag_rel` | 决定“有哪些食物可以被摇中, 每个食物带什么标签” |
| 用户偏好 | `user_preference` | 决定“喜欢什么, 避开什么, 预算大概多少” |
| 摇一摇结果 | `shake_history` | 记录“每次摇到了什么, 用户最后选没选, 文案是什么” |

### 每张表负责什么业务

#### `food_item` 是菜单主表

- 一行就是一个可推荐食物。
- `weight` 决定基础命中概率。
- `enabled` 用来控制食物是否还参与推荐。
- `reference_price` 先保留, 以后可以让预算真正参与计算。

#### `food_tag` 是标签字典表

- 存所有标签定义, 比如 `外卖`, `辣`, `清冰箱`。
- 系统模式本质上就是靠标签筛选实现。
- 偏好页里的喜欢标签和避免标签也都来自这里。

#### `food_item_tag_rel` 是食物和标签的关系表

- 一个食物会有多个标签, 所以不能把标签硬塞进一列字符串里。
- 例如 `黄焖鸡米饭` 可以同时关联 `外卖`, `米饭`, `辣`。

#### `user_preference` 是单用户偏好表

- 当前 V1 固定只有 `user_id = 1` 这一条。
- `favorite_tag_ids` 代表喜欢标签, 用于加权。
- `avoid_tag_ids` 代表避免标签, 用于排除候选。
- `last_mode` 用于首页恢复上一次选择的模式。

#### `shake_history` 是摇一摇流水表

- 每次摇一摇都插入一条记录。
- `result_status`
  - `PENDING`: 刚摇出来, 还没决定
  - `ACCEPTED`: 用户点了“就这样吃”
  - `REJECTED`: 用户点了“再摇一次”
- `food_name` 冗余保存是为了避免后续改菜名后, 历史列表显示错乱。

### 现在这套表怎么支撑业务

摇一摇时, 后端流程可以直接这样理解:

1. 从 `food_item` 取启用食物
2. 通过 `food_item_tag_rel` 和 `food_tag` 找出每个食物的标签
3. 读取 `user_preference` 做喜欢和避免过滤
4. 按模式标签和权重算出命中结果
5. 把结果写入 `shake_history`

如果你已经建过库, 再执行一次下面这份脚本补备注:
